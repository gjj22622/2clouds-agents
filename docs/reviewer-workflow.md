# Reviewer 品管閉環工作流程

本文件定義双云訓練平台中 **reviewer** 如何評估新人任務、做出通過或退修決定、撰寫 reviewer note，以及留下 trace log 的操作規範。

## 1. Reviewer 在訓練閉環中的位置

```
新人送出（submitted）
    ↓
Reviewer 評估
    ├── 通過 → reviewed（計入 60 分認證）
    └── 退修 → needs_revision（新人修改後可重新送出）
```

Reviewer 的任務不是打分數，而是讓新人知道「哪裡還不夠、為什麼不夠、如何改」。每一次退修都是訓練回饋，不是懲罰。

Phase 1 不實作獨立的「拒絕」（reject）狀態。需要完全撤銷任務指派時，由 admin 處理。

---

## 2. 品管四個面向

每份提交都從以下四個面向評估。**前三項是獨立的**；Final 是整合判斷。

### 2.1 Structure（結構完整性）

回答：**交付格式有沒有按要求完成？**

通過標準：
- 任務要求的每個輸出項目（如：前情提要、短文、檢查表）都有出現
- 格式符合任務說明（如：5 點以內、條列式）
- 不確定事項已標出，或已說明升級判斷

退修訊號：
- 缺少必要輸出項目
- 格式與任務說明不符
- 未標出不確定事項、未做升級判斷

### 2.2 Brand（品牌符合度）

回答：**內容是否符合品牌腦的語氣與規格？**

通過標準：
- 語氣與品牌腦定義的「語氣規範」吻合
- 用詞不在品牌禁忌清單內
- CTA 風格符合 channel rules
- 受眾定位正確，不過度承諾

退修訊號：
- 語氣偏口語、過度行銷、或違反品牌禁忌
- CTA 措辭與 channel rules 衝突
- 受眾定位模糊、或內容對象不明

### 2.3 Compliance（合規性）

回答：**有沒有觸犯應避免的承諾、法規風險或敏感議題？**

通過標準：
- 無誇大療效、保證成效等不當承諾
- 沒有觸及客戶合約未授權的議題
- 政治、宗教、競品等敏感類別未被提及或處理正確

退修訊號：
- 出現「保證」、「一定」、「最」等合規風險詞
- 提及客戶或競品的未授權敏感資訊
- 任何需要客戶確認才能發布的主張

> 合規問題比 brand 問題嚴重。若判定有合規風險，退修的同時應通知 Jacky。

### 2.4 Final（整合可交付）

回答：**這份交付物整體上能不能交給下一個流程節點？**

通過標準（全部滿足）：
- Structure、Brand、Compliance 均通過，或只有輕微品牌問題但不影響使用
- Trace log 格式正確、狀態變更有記錄
- 新人已做過自我品管（有 checklist 或說明）

退修標準（任一滿足）：
- 任何一個面向有明確問題
- 缺少 trace log 或狀態記錄
- 新人未做自我品管直接送出

## 3. 通過 vs. 退修：決策流程

```
1. 先看 Structure：格式完整嗎？
   → 有缺漏 → needs_revision
   → 完整，繼續

2. 看 Compliance：有合規風險嗎？
   → 有風險 → needs_revision（同時通知 Jacky）
   → 無風險，繼續

3. 看 Brand：語氣、用詞符合品牌腦嗎？
   → 明顯不符 → needs_revision
   → 輕微問題但不影響使用 → reviewed（在 note 留下建議）
   → 符合，繼續

4. Final 整合判斷：
   → 整體可交付 → reviewed
   → 有任何面向不過 → needs_revision
```

**原則：當你不確定要不要退修，偏向退修並附上具體理由。不明確的通過比不明確的退修更有害。**

## 4. 寫一份好的 reviewerNote

### 必備要素

一份有效的 reviewerNote 必須回答：
1. **哪個面向有問題**（structure / brand / compliance）
2. **具體的問題是什麼**（引用原文或格式）
3. **修改方向是什麼**（給出可執行的指引，不只指出錯誤）

### 格式建議

```
[問題面向] 具體問題描述。建議：修改方向。
```

### 不良示範（應避免）

❌ 「這篇文章不夠好，請重寫。」
- 原因：沒有指出具體問題，新人不知道從哪改起。

❌ 「語氣不對。」
- 原因：沒有說明哪裡不對、應該調整到什麼方向。

❌ 「請參考品牌腦。」
- 原因：太籠統，等同於沒有給指引。

### 良好示範

✅ 「[Brand] 第 2 段開頭「立刻獲得」屬於過度承諾用詞，觸犯品牌禁忌。建議改為「幫助你」或「協助品牌」，語氣保持積極但不誇大。」

✅ 「[Structure] 缺少升級判斷說明。任務要求列出不確定事項，請補上至少 1 個你覺得需要問 Jacky 或 Sophia 的問題。」

✅ 「[Brand] 語氣偏口語，與品牌腦定義的「專業但不冷硬」有落差。請對照 kg-brand-context，重新調整段落開頭與 CTA 用詞。CTA 建議參考品牌包的 channel rules 範例。」

## 5. Reviewer 的 Trace Log 責任

系統透過 `applyTaskStatusTransition` 自動記錄每次狀態轉換。Reviewer 的操作會產生以下 trace log：

| 操作 | action 值 | fromStatus | toStatus |
|---|---|---|---|
| 通過 | `review_created` | `submitted` | `reviewed` |
| 退修 | `revision_requested` | `submitted` | `needs_revision` |

Trace log 的 `actorId` 會記錄 reviewer 的 user ID，新人可以在任務的 trace 記錄中看到「誰在什麼時候做了什麼決定」。

**Reviewer 不需要手動填寫 trace log**，系統在操作時自動產生。Reviewer 只需要：
1. 執行狀態操作（reviewed 或 needs_revision）
2. 附上 reviewerNote（退修時必填）

## 6. 退件的訓練價值

needs_revision 狀態是訓練閉環中訓練價值最高的節點之一。

**為什麼退件比直接通過更有價值：**

- 新人在修稿過程中要對照品牌腦，練習「查腦」的動作
- 新人必須理解退件原因，練習「解讀品管意見」的能力
- 修稿後重新送出，再次通過的成就感比第一次直接通過更紮實

**Reviewer 的立場：**

退件不是評分失敗，而是教學介入。退件的同時，reviewerNote 就是教材。寫得越具體，訓練效果越好。

在 60 分認證規格中，明確要求：
- 3 次品管通過
- **1 次被退件後修正通過**（退件修稿循環是認證必要條件，不是扣分）

## 7. 各訓練 Stage 的 Reviewer 重點

| Stage | 代表任務 | Reviewer 重點 |
|---|---|---|
| 1 讀懂双云 | 整理三層架構摘要 | Structure：格式完整；邏輯：分工邊界有沒有說清楚 |
| 2 讀懂品牌腦 | 用品牌腦檢查貼文 | Brand：判斷是否正確；有沒有列出具體修正理由 |
| 3 內容任務 | 送出最小交付 | Structure + Brand + Compliance 全面評估；自我品管是否有做 |
| 4 品管與修稿 | 退件分析與修稿 | 看新人是否真正理解退件原因，修稿依據是否對應品牌腦 |
| 5 客戶服務 | 升級時機判斷 | 邏輯：升級判斷是否正確；不該升級的有沒有自己處理 |
| 6 60 分認證 | 完整模擬任務 | Final：整體可交付；trace log 完整；升級時機判斷準確 |
